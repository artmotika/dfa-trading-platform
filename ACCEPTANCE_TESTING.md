# План приемо-сдаточных испытаний (Acceptance Plan)

Этот план содержит сценарии тестирования для демонстрации соответствия системы регуляторным и функциональным требованиям.

## Сценарий 1: Регистрация и Идентификация (Identity)
**Требование:** Ведение реестра владельцев ЦФА через достоверную идентификацию.
**Запрос:**
```http
POST http://localhost:8083/api/v1/auth/esia/login?code=thesis-test-123
```
**Ожидаемый результат:**
- Система создает пользователя с признаком `kycStatus = APPROVED`.
- В ответ приходит JWT-токен.
- В `auth-service` логируется событие `users.registered`.

---

## Сценарий 2: Контроль лимитов инвестора (Regulatory Limit)
**Требование:** Соблюдение лимита покупок в 600 000 руб. для неквалифицированных инвесторов.
**Запрос:**
```http
POST http://localhost:8080/api/v1/orders
{
  "userId": "uuid-пользователя-esia",
  "assetId": "asset-1",
  "type": "BUY",
  "amount": 7,
  "price": 100000
}
```
**Ожидаемый результат:**
- Система возвращает **HTTP 500/400** с сообщением: `Retail investor annual limit (600,000 RUB) exceeded`.
- Ордер блокируется на этапе шлюза (API Gateway).

---

## Сценарий 3: Полный цикл сделки (End-to-End Trade)
**Требование:** Исполнение ордера, запись в реестр и подтверждение в блокчейне.
**Запрос:**
```http
POST http://localhost:8080/api/v1/orders
{
  "userId": "uuid-пользователя-esia",
  "assetId": "asset-1",
  "type": "BUY",
  "amount": 1,
  "price": 50000
}
```
**Ожидаемый результат:**
1. Шлюз пропускает ордер (лимит не превышен).
2. Движок сохраняет ордер со статусом `PENDING`.
3. Коннектор отправляет транзакцию в Solana Devnet.
4. После подтверждения статус меняется на `COMPLETED`.
5. В таблице `trades_ledger` (БД PostgreSQL) появляется запись с `tx_hash` из блокчейна.

---

## Сценарий 4: Налоговый агент (Tax Reporting)
**Требование:** Расчет НДФЛ при реализации ЦФА.
**Запрос:**
```http
POST http://localhost:8080/api/v1/orders
{
  "userId": "uuid-пользователя-esia",
  "assetId": "asset-1",
  "type": "SELL",
  "amount": 1,
  "price": 60000
}
```
**Ожидаемый результат:**
- После исполнения сделки в таблице `tax_ledger` (PostgreSQL) появляется запись.
- Сумма налога составляет **7 800** (60 000 * 0.13).

---

## Сценарий 5: Корпоративные действия (Dividends)
**Требование:** Автоматизация выплат по ЦФА.
**Действие:**
Используйте **Admin Panel** (`admin-ui/index.html`) или отправьте запрос к `trading-engine`.
**Ожидаемый результат:**
- Система рассчитывает баланс пользователя на основе истории сделок.
- В Kafka отправляется событие `dividend.payout`.
- Коннектор выполняет транзакцию выплаты в блокчейне.

---

## Сценарий 6: AML Контроль (Security)
**Требование:** Защита от подозрительных операций и подозрительных объемов.
**Запрос:**
Отправьте покупку на сумму **2 000 000 руб.** за раз.
**Ожидаемый результат:**
- Отказ по причине `AML Policy Violation: Volume limit exceeded`.
- В базе данных у пользователя повышается риск-скор (`amlRiskScore += 50`).
